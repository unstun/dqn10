# 2026-04-10 调研记录

## 01:30 — LLM 本地知识缓存系统调研（27 方案 / 5 方向）

- **背景**：Dr Sun 提出 LLM 幻觉严重问题，设计三层检索架构（L1本地缓存/L2联网搜索/L3参数知识），当前工作流缺少 L1 层，需系统摸底方案后决策
- **来源**：联网调研 + arXiv，主要论文标识与 URL 见下

### 方向 A — 判定与幻觉检测（4 items）

- **A1 Self-RAG**：arXiv:2310.11511；检索+反思令牌，允许模型动态决策是否调用检索，TruthfulQA 提升 10.9 pts
- **A2 CRAG**：Corrective RAG，基于相关性分数切换 Web 搜索，显著减少不可靠检索带来的幻觉
- **A3 Rowen**：arXiv:2402.10612（SIGIR-AP 2025）；TruthfulQA +16.74%，使用一致性票决机制检测不一致输出
- **A4 MiniCheck**：arXiv:2404.10774；770M 参数达 GPT-4 级事实核查精度，成本降低 400×，适合 /archive 流程质量门控

### 方向 B — 语义缓存与向量数据库（6 items）

- **B1 GPTCache**：基于语义相似度缓存 LLM 响应，完全命中时 0 API 调用；适合高频重复查询场景
- **B2 Amazon Bedrock Verified Semantic Cache**：aws-samples GitHub（https://github.com/aws-samples/amazon-bedrock-verified-semantic-cache）；加入事实验证层，拒绝缓存幻觉响应
- **B3 FAISS / Milvus / Weaviate / Chroma**：向量数据库方案；对本项目属 overkill，无需集群部署
- **B4 txtai**：GitHub: neuml/txtai；轻量全栈 NLP（Embeddings + LLM + Pipelines），单文件部署，适合本地小型知识库
- **B5 RouteRAG**：arXiv:2512.09487；RL 驱动检索路由，动态决策何时用缓存/联网/LLM参数，多跳问答 +8%
- **B6 Krites (Apple)**：arXiv:2602.13165；设备端语义缓存，覆盖率 +136%~290%，推理速度 2-4×，仅适合苹果设备

### 方向 C — 持久化记忆与知识图谱（6 items）

- **C1 Zep/Graphiti**：arXiv:2501.13956；时序知识图谱，支持增量更新，Deep Memory retrieval 精度 +94.4%
- **C2 Mem0**：arXiv:2504.19413；分层记忆系统，token 消耗减少 90%，适合多会话长周期项目
- **C3 LangGraph Persistence**：LangGraph 官方持久化层，基于 checkpoints，与 Python 生态深度集成
- **C4 Microsoft GraphRAG**：arXiv:2404.16130；社区摘要图 RAG，全局语义查询优于向量检索，但资源消耗大
- **C5 ConPort**：GitHub: iyuangang/conport；MCP-based 会话知识管理，与 Claude Code 原生集成，最契合当前架构
- **C6 txtai**：（同 B4，轻量替代 ConPort，单进程部署）

### 方向 D — 检索增强生成优化（6 items）

- **D1 Anthropic Contextual Retrieval**：Anthropic 官方博客（https://www.anthropic.com/research/contextual-retrieval）；为每个 chunk 追加上下文前缀，检索失败率降低 35-67%；结合 BM25 可降低 49%；1-2 天可集成
- **D2 A-RAG 分层检索**：Adaptive RAG；对本项目可复用现有 ACE + Grep 两层，低成本实现分层检索
- **D3 HyDE**：假设文档嵌入（Hypothetical Document Embeddings）；生成假设答案后做嵌入检索，提升召回
- **D4 FLARE**：前向主动检索，仅在生成概率低时触发检索，减少不必要检索调用
- **D5 RAGatouille / ColBERT**：token 级晚期交互检索，精度优于双编码器，但部署复杂
- **D6 Self-RAG Fine-tuning**：在项目语料上微调 Self-RAG；资源消耗大，当前阶段不适合

### 方向 E — 现有基础架构分析（5 items）

- **E1 bigmemory 热区/冷区分层**：已是知识缓存雏形；主要缺口：热区无索引、memory-worker 每次重读全部文件
- **E2 .pipeline/ 长期知识库**：文献库/实验台账已结构化；缺口：无语义检索，依赖 Grep 精确匹配
- **E3 ACE（Augment Context Engine）**：MCP 工具，已集成语义搜索；缺口：无缓存层，每次重新计算嵌入
- **E4 memory-worker 效率**：当前每次调用读取全量 bigmemory 文件；优化方向：增加 TTL + 摘要索引
- **E5 Harness 整体评估**：bigmemory + .pipeline/ + ACE 三者已覆盖基本 L1 功能，主要缺口在缓存命中率和检索精度

### 关键结论

- **优先实施建议**（按 ROI 排序）：
  1. **E2 Anthropic Contextual Retrieval**：检索失败率降 35-67%，1-2 天，最高 ROI
  2. **E1 A-RAG 分层检索**：复用现有 ACE + Grep，1-2 天，低改造成本
  3. **D1 MiniCheck**：/archive 流程质量门控，0.5-1 天，770M 参数本地可运行
  4. **C5 ConPort 或 C6 txtai**：更大架构升级，3-4 天，替换 memory-worker 读取全量文件的方式
- **排除方案**：LangChain/向量数据库集群/GraphRAG — 对当前规模属 overkill，维护成本过高
- **最契合架构**：MCP-based 本地知识缓存服务（ConPort 路线），与 Claude Code 原生集成，无需额外服务进程
