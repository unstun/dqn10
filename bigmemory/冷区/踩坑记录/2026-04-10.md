# 2026-04-10 踩坑

## Factory/Droid JSONL 格式与 Claude Code 差异（逆向工程）

- **现象**：dump_conversation.py 用 Claude Code 的 JSONL 解析逻辑去读 Factory/Droid session 文件，全部解析失败（找不到 user/assistant 消息）
- **原因**：两个平台的 JSONL 格式存在 5 处结构性差异：
  1. **消息类型**：Droid 顶层 `type="message"` + `message.role` 区分角色；Claude Code 顶层 `type="user"` / `type="assistant"` 直接标角色
  2. **首行标记**：Droid 第一行 `type="session_start"`（含 id/title/owner/cwd）；Claude Code 第一行 `type="summary"` 或 queue-operation
  3. **用户真实输入定位**：Droid 用户消息的 content blocks 列表中，真实输入在末尾（前面塞满 system-reminder）；Claude Code 靠 `isMeta` 标记区分
  4. **subagent 存储**：Factory subagent 无 `<session_id>/subagents/` 子目录，每个 subagent 是独立 session 文件；Claude Code 在主 session 下有 subagents/ 子目录
  5. **session 路径**：`~/.factory/sessions/<project-hash>/` vs `~/.claude/projects/<project-hash>/`
- **解决**：在 dump_conversation.py 中新增自动格式检测 + 独立的 Droid 解析分支，用首行 type 字段区分格式
- **预防**：解析外部 AI 平台 session 文件时，先逆向确认格式再写解析器，不要假设格式一致；两个平台的内部数据结构完全独立演化
