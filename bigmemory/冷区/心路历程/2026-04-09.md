# 2026-04-09 心路

## bigmemory 设计三次迭代的教训
从 v1(hook sub-agent 全自动)→ v2(纯 bash dump)→ v3(按需拉取主 AI 操作)→ 最终版(自动派 worker + 手动归档),每次迭代都是因为 Dr Sun 指出了一个我没考虑到的用户视角问题:
1. "sub-agent 操作人看不到" — 透明性比自动化更重要
2. "直接 dump 是一坨屎山" — 需要智能过滤,不能无脑注入
3. "应该用户第一句话后自动抓,结束时我自己手动归档" — 入口自动,出口手动,给用户控制权

核心认知:记忆系统的设计不是技术问题,是**谁有控制权**的问题。入口可以自动(因为只是读取),出口必须手动(因为涉及写入和判断什么值得保留)。

## Claude Code 与 Droid 统一的局限
一开始以为同步文件就能统一体验,但 Dr Sun 反复提醒"不在这"。最终认清:
- **可统一的**: commands、skills、droids/agents、hooks、settings.json、CLAUDE.md≡AGENTS.md
- **不可统一的**: /compact、内置 memory、context 可视化、平台 UI 行为

这不是配置能解决的问题,是两个平台的产品决策不同。

## 为什么决定用 symlink 合并 .claude/ 和 .factory/ 的 commands/skills

Harness 评审发现 `.claude/commands/`(10 个)和 `.factory/commands/`(10 个)内容完全一致，`.claude/skills/`(37 个)和 `.factory/skills/`(37 个)也完全一致。两份副本分开维护，每次改动都要同步两处，这是典型的重复维护陷阱。

决策逻辑：以 `.claude/` 为单一来源(source of truth)，`.factory/commands/` 和 `.factory/skills/` 改为指向 `.claude/` 的 symlink。这样 Claude Code 端改一次，Factory/Droid 端自动获得更新，彻底消除双份维护成本。rsync 到远端时加 `--copy-links` 把 symlink 展开为真实文件，保持远端可用。

接受 symlink 隐式引用的理由：Harness 层的 commands/skills 由 AI 消费，不由人逐行读，symlink 的间接性对 AI 透明。只要文档层面说明来源，维护者不会困惑。

## 为什么 agents 和 droids 不合并

`.claude/agents/`(5 个)和 `.factory/droids/`(6 个)格式不同：

- `.claude/agents/`：纯 prompt 文本，无 frontmatter，Claude Code 原生 agent 格式
- `.factory/droids/`：有 YAML frontmatter（model、role、tools 字段）+ 精简版 prompt，Factory 专有格式

除格式外，职责也不同：droids 是 Factory 调度的 gpt-5.4 工人，agents 是 Claude Code 的子 agent。两者平台不同、模型不同、调度方式不同。强行合并会破坏格式约定，得不偿失。

## 36 条硬规则的分层建议

评审时发现 36 条规则混在一起，认知负荷高，AI 解析时也难以快速定位优先级。建议分三层：

1. **通用工作流层**（约 10 条）：适用于所有项目的基本纪律，如"改文件前计划 + 等确认"、"每次 commit 前 git push 备份"、"不确定先问"
2. **领域专用层**（约 15 条）：DQN/RL 论文写作专用，如"公式用标准记法"、"禁止 enumerate 列表"、"SS-RRT* 不泄漏 cost-to-go"
3. **踩坑防护层**（约 11 条）：从具体失败经验提炼，如"WebFetch 与 WebSearch 不并行"、"conda run 必须 --cwd"

这次没有立即重构，因为规则已经是 AI 可执行状态，分层只是提升可读性，不紧急。记录在这里供下次 Harness 维护时参考。

## scripts/ 重组与未关闭决策清理——精简比功能更难

Harness 得了 8/10 分，两个扣分点都落在"结构混乱"而非"缺功能"。这说明 Harness 到了一个新阶段：**功能基本完备，瓶颈转向可维护性**。

scripts/ 平铺 42 个文件，问题不在于数量，在于使用者（AI）无法通过文件名快速判断意图。`run_xxx.sh` 是启动脚本还是分析脚本？`auto_pipeline_cnn_dqn.sh` 是一次性的还是长期维护的？平铺结构迫使每次都要全量扫描。3 层分类（run/ analysis/ 顶层单文件）解决的是**认知负载**问题，不是磁盘问题。

deploy_server.py 单独保留顶层这个决定值得记录：它是跨多个 profile 的统一部署入口，既不是"运行实验"也不是"分析结果"，放入任何子目录都会产生误导。顶层保留是语义正确的选择。

未关闭决策的清理也有意思：6 条 open 中，5 条其实早已有答案，只是没有人触发关闭动作。这揭示了一个元问题——**决策跟踪系统需要定期清扫**，否则 open 列表会因惰性积累而失去参考价值。以后每次 Harness 维护会话都应该主动过一遍未关闭决策。

.agents/ 目录的删除是这次最干净的决策：190+ 文件，全部冗余，因为确认只用 Claude Code 和 Droid 两个平台。保留它的唯一理由是"万一以后用"，但这种心理是熵增的来源。删掉。

## 为什么把 DQN10 项目结构改成三阶段分层

DQN9 的根本问题是：规划、实验、写作三件事挤在同一个扁平目录里，思路边界不清晰，经常在一个会话里串联多个阶段。代码改了半天，突然要写论文；文献没读完，就开始调参。这不是效率问题，是认知架构问题。

三阶段（1_survey → 2_experiment → 3_paper）强制了一个单向的信息流：

- **1_survey** 只做"我们知道什么"——文献综述、调研笔记、研究问题聚焦
- **2_experiment** 只做"我们做了什么"——代码、训练、数据整理
- **3_paper** 只做"我们怎么说"——论文写作、图表、报告

每个阶段有明确的输入（上一阶段产物）和输出（交付给下一阶段）。会话协议里"每次只做一件事"现在有了物理上的支撑——目录本身就在告诉你当前在哪个阶段。

关于把"实验"和"数据分析"合并为同一阶段：这是合理简化。分析本质上是实验的延伸，把跑出来的数据整理成结论，不值得单独一个目录。如果分析工作量变大，可以在 `2_experiment/` 内用子目录区分，不需要再提升一层。

这次重组花了很多时间（1355 文件 rename），但代价是值得的——架构清晰后每次开会话的认知负荷都会降低。
