# 2026-04-10 心路历程

## 为什么从 22 项收窄到 3 方向再扩展到 5 方向 27 项

初始 outline 生成了 22 个 items，Dr Sun 的质疑点是"哪个好"——这让我意识到：数量不等于质量，在没有筛选框架的情况下调研 22 项只会产生 22 份孤立报告，不能回答"应该选哪个"这个核心问题。

第一次收窄到 3 个方向（判定机制/缓存工程/现有基础）是为了先建立判断框架，再决定是否需要更多信息。核心逻辑：**先问"我们缺什么"，再问"有哪些方案能补"**。

收窄评估后发现：
1. 现有架构（bigmemory + .pipeline/ + ACE）已有 L1 缓存雏形，不是从零开始
2. 主要缺口集中在检索精度（chunk 无上下文）和缓存命中率（每次重读全量）
3. 排除了向量数据库集群路线（运维成本过高）

在这个判断基础上，扩展为 5 方向 27 项是为了覆盖"如何补缺口"的完整解决方案空间——既要看新方案（方向 A-D），也要精确评估现有基础能走多远（方向 E）。

## 为什么选 MCP 本地缓存而非 LangChain / 向量数据库方案

LangChain 是全栈框架，引入后几乎每个模块都要重写（retriever / memory / chain），对当前以 Claude Code + MCP 为核心的架构属于破坏性替换，不是增量改进。

向量数据库集群（Milvus / Weaviate）的问题不在能力，在规模匹配：DQN10 的知识库规模（bigmemory 几十个文件，.pipeline/ 几百 KB 文本）不需要分布式向量检索，txtai 单文件部署足够。

MCP 本地缓存服务（ConPort 路线）的优势：
- 与 Claude Code 原生集成，无需改变主工作流
- MCP 是 harness 已有的工具调用范式，学习曲线为零
- 可以渐进迁移：先加 ConPort，再逐步把 memory-worker 的文件读取替换为 MCP 调用

## 为什么优先 E2 Contextual Retrieval + E1 A-RAG + D1 MiniCheck 而非直接上 ConPort / GraphRAG

ConPort 是架构级替换，ROI 更高但改造成本也更高（3-4 天 + 需要迁移现有 bigmemory 结构）。在没有验证"检索精度是否真的是瓶颈"之前，贸然替换架构风险过高。

Contextual Retrieval 是纯加法：在现有 chunk 切分流程里多加一步"为每个 chunk 生成上下文前缀"，不改变任何现有接口，1-2 天可见效，且 Anthropic 官方数据显示检索失败率降低 35-67%，是迄今最高置信度的单点改进。

MiniCheck 的优先级来自实用性：/archive 流程目前没有质量门控，归档的内容质量完全依赖归档 agent 的判断。770M 参数本地可运行，0.5-1 天集成，立刻给整个 bigmemory 增加一层事实验证护栏。

**总体逻辑**：先用低成本改进验证问题是否存在（Contextual Retrieval + MiniCheck），再决定是否值得做架构级升级（ConPort）。避免在不清楚瓶颈在哪里的情况下过度工程化。
