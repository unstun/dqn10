# 论文写作踩坑记录与规范

> 本文件记录论文写作中已发现并修正的错误模式，供后续写作时对照检查。
> 被 `CLAUDE.md` 引用，每次对话自动加载。

---

## 规则 1：禁止捏造术语

**错误示例**：
- "战略--战术分离原则"——文献中不存在此术语，是 AI 自创的概念名称

**正确做法**：
- 用朴素语言描述工程设计选择，不要包装成"XX原则""XX范式""XX机制"等听起来像专有名词的说法
- 如果要命名，必须能指向具体文献出处

**修正对照**：
| 错误 | 修正 |
|------|------|
| "遵循**战略--战术分离原则**：卷积网络…承担*战略方向选择*的职责" | "将路径引导与碰撞回避分配至两条独立的数据路径：卷积网络…" |

---

## 规则 2：禁止过度包装简单概念

**错误示例**：
- "由低维标量特征向量与多通道地图张量**耦合而成的二元组**" → 其实就是"标量特征 + 地图的组合"
- "特征级拼接融合（Feature Concatenation）" → 就是"拼接"，给 concat 加括号注释多余
- "四个核心模块级联构成" → "四部分组成"
- "两类异构输入" → "两类输入"
- "对自身的运动状态与周围环境的空间拓扑结构进行**同步感知**" → "同时获取自身运动状态与环境空间信息"

**判断标准**：如果去掉修饰词后意思不变，就应该去掉。

**修正对照**：
| 错误 | 修正 |
|------|------|
| "由低维标量特征向量与多通道地图张量耦合而成的二元组" | "标量特征向量与多通道地图张量的组合" |
| "特征级拼接融合（Feature Concatenation），馈入全连接层" | "拼接后，馈入全连接层" |
| "四个核心模块级联构成" | "四部分组成" |

---

## 规则 3：术语必须与原文献一致

**错误示例**：
- 将 Dueling Network Architecture（Wang et al., 2016）的标题写成"Dueling 价值分解"——"Value Decomposition"是 MARL 领域 VDN/QMIX 的术语，张冠李戴

**正确做法**：
- 使用原论文的标准名称
- 如需翻译，查阅该领域已有的中文翻译惯例

**修正对照**：
| 错误 | 修正 |
|------|------|
| "Dueling 价值分解" | "Dueling 网络架构" |

---

## 规则 4：避免推销性/修饰性语言

**错误模式与修正**：
| 错误 | 修正 | 原因 |
|------|------|------|
| "显式编码了…最短测地线距离" | "编码了…测地线距离" | "显式"多余，编码方式已由公式说明；"最短"是"测地线"固有含义 |
| "在极大程度上缓解了…" | "为…提供稠密方向引导" | 程度断言需要实验证据 |
| "显式地保留了系统对动态安全边界的敏锐感知能力" | "保留障碍物距离信息" | 拟人化修饰 |
| "突破了常规的…范式" | "采用…而非…" | "突破"是评价词，应由审稿人判断 |
| "旨在引导…" | "引导…" | "旨在"是废话 |

**高频修饰词检查清单**（写完后全文搜索）：
- "显式" → 大部分可删
- "旨在" → 全部删
- "突破" → 改为客观陈述
- "极大程度" → 删除或用实验数据支撑
- "完全/完整/精确/精准" → 审视是否必要

## 规则 5：段落长度控制

- 一个段落只讲一个论点
- 超过 5 行的段落检查是否可拆分
- 避免在一句话中塞入 3 个以上从句
